# 시험 문제 생성 규칙 (RULE)

이 문서는 **시험 문제 파일을 어떤 형식으로 만들어야 하는지** 정의합니다.
AI 또는 사람이 이 문서를 그대로 따르면, 만든 파일을 사이트에 넣었을 때 곧바로 시험으로 변환됩니다.
문제와 선택지에는 답을 유추할 수 있게끔 개념을 넣지 않습니다. 오로지 사용자의 능력으로만 유추하여 풀 수 있게 만듭니다.
또한 해설에는 해당 개념을 쉽게 외우는 팁을 붙이고, 각 선택지에는 정답 제출 시 왜 맞고 틀렸는지 알 수 있도록 보기 해설을 무조건 넣습니다.
하나의 문제 파일에서 만들었던 문제와 비슷한 문제를 만들지 않습니다.(다른 파일은 상관 없습니다.)
문제의 주석을 보고 정답을 맞춰야하는 문제 같은 경우에는 확실하게 주석을 명시합니다.(절대 주석이 없게 만들지 않습니다.)

지원 형식은 두 가지입니다.

- **TXT 형식** — 사람이 손으로 쓰기 쉬움 (`.txt`)
- **JS 형식** — 구조가 명확하고 보기별 해설까지 표현하기 좋음 (`.js`)

문제 유형은 **객관식**, **주관식**, **연결형** 세 가지입니다.

---

## 1. TXT 형식

### 1-1. 권장(인라인) 형식

각 문제는 **빈 줄**로 구분합니다. 정답과 해설을 문제 바로 아래에 적습니다.

#### 객관식

```
Q1. 질문 내용을 여기에 적습니다.
① 첫 번째 보기
② 두 번째 보기
③ 세 번째 보기
④ 네 번째 보기
정답: ③
해설: 왜 ③번이 정답인지 설명합니다.
보기해설:
① 이 보기가 틀린 이유
② 이 보기가 틀린 이유
③ 이 보기가 정답인 이유
④ 이 보기가 틀린 이유
```

- 문제 번호는 `Q1.` `1.` `1)` 모두 인식합니다.
- 보기 기호는 `①②③④…` 또는 `1) 2) 3)` 형식을 사용합니다.
- `정답:` 에는 기호(`③`) 또는 숫자(`3`)를 적습니다.
- `해설:` 은 무조건 포함합니다. 정답 개념과 함께 쉽게 외우는 팁을 적습니다.
- `보기해설:` 도 무조건 포함합니다. 각 보기마다 왜 맞고 틀린지를 적으면 보기 아래에 표시됩니다.

#### 주관식

```
[주관식] 질문 내용을 여기에 적습니다.
모범답안: 모범 답안 내용을 적습니다. 여러 줄도 가능합니다.
해설: 채점 시 참고할 핵심 키워드나 보충 설명.
```

- 주관식은 맨 앞에 `[주관식]` 을 붙이거나, `모범답안:` 줄이 있으면 자동으로 주관식으로 인식됩니다.
- 보기(①②③④)가 없는 문제도 자동으로 주관식 처리됩니다.

#### 연결형

맨 앞에 `[연결형]` 을 붙이고, **왼쪽 = 오른쪽** 형태로 *정답 짝*을 한 줄에 하나씩 적습니다.

```
[연결형] 다음 디자인 패턴과 핵심 의도를 연결하세요.
싱글톤 = 인스턴스를 하나만 보장한다
옵저버 = 상태 변화를 자동 통지한다
팩토리 = 객체 생성을 캡슐화한다
해설: 각 패턴이 무엇을 해결하려는지 한 단어로 떠올리면 쉽습니다.
```

- 구분자는 `=` 외에 `->`, `→`, `::` 도 인식합니다.
- **적은 줄이 곧 정답 짝**입니다. 실제 화면에서는 답이 대각선으로 드러나지 않도록 오른쪽 보기 순서가 자동으로 섞여 표시됩니다.
- 풀이 화면에서는 각 왼쪽 항목 옆 **드롭다운**에서 오른쪽 보기를 고릅니다.
- 채점은 **한 문제의 모든 짝을 맞혀야** 정답 1개로 집계됩니다(부분 점수 없음).
- `해설:` 은 필수 사항입니다(`정답:` 줄은 필요 없습니다).

### 1-2. 분리(레거시) 형식 — 기존 `문제.txt` 호환

문제를 모두 먼저 쓰고, 맨 아래에 `정답 및 해설` 구분선을 두는 형식도 지원합니다.

```
Q1. 질문...
① 보기  ② 보기  ③ 보기  ④ 보기

Q2. 질문...
...

=========================================================
정답 및 해설
=========================================================

Q1. 정답: ③
해설: 설명...
암기 팁: ...

Q2. 정답: ①
해설: 설명...
```

- `정답 및 해설` 이라는 문구가 들어간 줄을 기준으로 문제부와 정답부를 나눕니다.
- 정답부의 `Q1.`, `Q2.` 번호 순서대로 위 문제와 1:1로 연결됩니다.
- `암기 팁:` 은 해설에 함께 표시됩니다.

> ⚠️ 레거시 형식은 객관식만 표현할 수 있습니다. 주관식이나 보기별 해설이 필요하면 **인라인 형식이나 JS 형식**을 사용하세요.

## 2. JS 형식 (권장)

전역 배열 `QUIZ_QUESTIONS` 를 정의합니다. 가장 안전한 방법은 `window.QUIZ_QUESTIONS` 입니다.

```js
window.QUIZ_QUESTIONS = [
  // ── 객관식 ──
  {
    type: "multiple",
    question: "질문 내용",
    choices: ["보기1", "보기2", "보기3", "보기4"],
    answer: 3,                    // 정답 보기 번호 (1부터 시작: 1=①, 2=②, 3=③, 4=④)
    explanation: "전체 해설",
    choiceExplanations: [
      "보기1이 틀린 이유",
      "보기2가 틀린 이유",
      "보기3이 정답인 이유",
      "보기4가 틀린 이유"
    ]
  },

  // ── 주관식 ──
  {
    type: "short",
    question: "질문 내용",
    modelAnswer: "모범 답안 (여러 줄 가능)",
    explanation: "채점 참고용 핵심 키워드"
  },

  // ── 연결형 ──
  {
    type: "matching",
    question: "다음 디자인 패턴과 핵심 의도를 연결하세요.",
    pairs: [                       // [왼쪽, 정답 오른쪽] — 정답 짝을 그대로 적습니다
      ["싱글톤", "인스턴스를 하나만 보장한다"],
      ["옵저버", "상태 변화를 자동 통지한다"],
      ["팩토리", "객체 생성을 캡슐화한다"]
    ],
    explanation: "각 패턴이 무엇을 해결하려는지 떠올리면 쉽습니다." // 연결형도 해설은 필수
  }
];
```

### 필드 설명

| 필드 | 유형 | 필수 | 설명 |
|------|------|:----:|------|
| `type` | `"multiple"` / `"short"` / `"matching"` | 권장 | 생략 시 `pairs`가 있으면 연결형, `choices`가 있으면 객관식, 없으면 주관식 |
| `question` | 문자열 | ✅ | 문제 본문 |
| `choices` | 문자열 배열 | 객관식 ✅ | 보기 목록 |
| `answer` | 숫자 | 객관식 ✅ | 정답 보기 번호. **1부터 시작** (`1`=첫 번째 보기 ①) |
| `explanation` | 문자열 | ✅ | 전체 해설 + 외우는 팁 |
| `choiceExplanations` | 문자열 배열 | 객관식 ✅ | 보기 순서대로 1:1, 각 보기 아래 표시 |
| `modelAnswer` | 문자열 | 주관식 ✅ | 모범 답안 |
| `pairs` | `[왼쪽, 오른쪽]` 배열 | 연결형 ✅ | **정답 짝** 목록. 화면에선 오른쪽 보기가 자동으로 섞여 표시됨 |

## 3. 동작 방식 요약

- **한 페이지에 한 문제씩** 표시됩니다.
- **객관식**: 보기를 선택하면 즉시 정답/오답이 색으로 표시되고, 각 보기 아래에 (있다면) 해설이 나옵니다.
- **주관식**: 답을 작성해 제출하면 모범 답안이 나오고, 사용자가 직접 `⭕ 맞음 / ❌ 틀림` 으로 채점합니다.
  - 답을 **제출했다면 자가 채점(⭕/❌)을 선택해야** 다음 문제·결과로 넘어갈 수 있습니다. (제출하지 않고 비워 두면 그냥 넘어갈 수 있습니다.)
- **연결형**: 각 왼쪽 항목 옆 드롭다운에서 오른쪽 보기를 고른 뒤 제출하면 짝마다 `⭕/❌` 가 표시됩니다. **모든 짝을 맞혀야** 정답 1개로 집계됩니다.
- 유형 배지(`객관식`·`주관식`·`연결형`)는 문제 번호(`Q1.`) 바로 옆에 표시됩니다.
- 모든 문제를 풀면 **`정답 수 / 전체 문항 수` 와 정답률(%)** 이 표시됩니다.
  - 객관식은 선택한 답으로, 주관식은 자가 채점 결과로, 연결형은 전체 짝 일치 여부로 집계됩니다.

## 5. 완성 예시 (그대로 복사해서 시작하기)

아래 두 예시는 **실제 내용이 채워진 완전한 파일 한 개**입니다.
같은 5문항(객관식 2 + 주관식 2 + 연결형 1)을 TXT와 JS 두 형식으로 똑같이 표현했습니다.
새 문제를 만들 때는 아래 예시를 통째로 복사한 뒤 내용만 바꾸세요.

### 5-1. TXT 완성 예시 (`예시.txt`)

```
Q1. 다음 중 운영체제의 핵심 기능이 아닌 것은?
① 프로세스 관리
② 메모리 관리
③ 소스 코드 컴파일
④ 파일 시스템 관리
정답: ③
해설: 컴파일은 컴파일러(개발 도구)의 역할이며, 운영체제의 핵심 기능이 아닙니다.
보기해설:
① 프로세스 생성·스케줄링은 운영체제의 핵심 기능입니다.
② 메모리 할당·회수는 운영체제가 담당합니다.
③ 소스 코드 컴파일은 컴파일러가 하는 일입니다.
④ 파일 저장·접근 관리는 운영체제의 핵심 기능입니다.

Q2. 교착 상태(Deadlock) 발생의 4가지 필요조건이 아닌 것은?
① 상호 배제
② 점유와 대기
③ 선점 가능
④ 환형 대기
정답: 3
해설: 교착 상태의 조건은 비선점(점유한 자원을 강제로 빼앗을 수 없음)입니다. "선점 가능"은 오히려 교착 상태를 깨는 조건입니다.
보기해설:
① 상호 배제는 한 자원을 한 번에 한 프로세스만 쓸 수 있다는 조건으로, 교착 상태의 필요조건입니다.
② 점유와 대기는 자원을 쥔 채 다른 자원을 기다리는 조건으로, 필요조건입니다.
③ 교착 상태의 조건은 "비선점"입니다. 선점이 가능하면 교착을 깰 수 있으므로 필요조건이 아닙니다.
④ 환형 대기는 프로세스들이 원형으로 서로의 자원을 기다리는 조건으로, 필요조건입니다.

Q3. [주관식] 프로세스와 스레드의 차이를 자원 공유 관점에서 한 문장으로 설명하세요.
모범답안: 프로세스는 독립된 메모리 공간을 가지지만, 스레드는 같은 프로세스 안에서 코드·데이터·힙 영역을 공유한다.
해설: '독립된 메모리'와 '메모리 공유'라는 두 키워드가 들어가면 정답으로 인정합니다.

Q4. [주관식] 캐시 메모리를 사용하는 이유를 '지역성' 개념과 함께 설명하세요.
모범답안: CPU와 주기억장치의 속도 차이를 줄이기 위해, 최근 참조한 데이터가 다시 참조될 가능성이 높다는 지역성(시간·공간 지역성)을 이용해 자주 쓰는 데이터를 빠른 캐시에 두기 때문이다.
해설: 핵심 키워드 - 속도 차이, 시간 지역성, 공간 지역성.

Q5. [연결형] 다음 CPU 스케줄링 알고리즘과 핵심 특징을 연결하세요.
FCFS = 먼저 도착한 순서대로 처리한다
SJF = 실행 시간이 가장 짧은 작업을 먼저 처리한다
RR = 정해진 시간 할당량만큼 돌아가며 처리한다
해설: 약자를 풀어 보면 특징이 그대로 드러납니다 - First Come First Served, Shortest Job First, Round Robin.
```

> 포인트
> - 문제끼리는 **빈 줄**로 구분합니다.
> - 객관식 `정답:` 은 기호(`③`)와 숫자(`3`) 모두 됩니다(Q1·Q2 비교).
> - 객관식은 `해설:` 과 `보기해설:` 을 **모두 필수**로 적습니다(Q1·Q2 둘 다 포함).
> - 주관식은 `[주관식]` 라벨 또는 `모범답안:` 줄로 인식되며, `해설:` 도 필수입니다.
> - 연결형은 `[연결형]` 라벨을 붙이고 정답 짝을 한 줄씩 적으며, `해설:` 도 필수입니다(Q5).

### 5-2. JS 완성 예시 (`예시.js`)

위 TXT와 **완전히 동일한 5문항**을 JS로 표현한 것입니다.

```js
window.QUIZ_QUESTIONS = [
  // ── Q1. 객관식 (보기별 해설 포함) ──
  {
    type: "multiple",
    question: "다음 중 운영체제의 핵심 기능이 아닌 것은?",
    choices: ["프로세스 관리", "메모리 관리", "소스 코드 컴파일", "파일 시스템 관리"],
    answer: 3, // 1부터 시작 → 3 = ③ "소스 코드 컴파일"
    explanation: "컴파일은 컴파일러(개발 도구)의 역할이며, 운영체제의 핵심 기능이 아닙니다.",
    choiceExplanations: [
      "프로세스 생성·스케줄링은 운영체제의 핵심 기능입니다.",
      "메모리 할당·회수는 운영체제가 담당합니다.",
      "소스 코드 컴파일은 컴파일러가 하는 일입니다.",
      "파일 저장·접근 관리는 운영체제의 핵심 기능입니다."
    ]
  },

  // ── Q2. 객관식 ──
  {
    type: "multiple",
    question: "교착 상태(Deadlock) 발생의 4가지 필요조건이 아닌 것은?",
    choices: ["상호 배제", "점유와 대기", "선점 가능", "환형 대기"],
    answer: 3, // 3 = ③ "선점 가능"
```
